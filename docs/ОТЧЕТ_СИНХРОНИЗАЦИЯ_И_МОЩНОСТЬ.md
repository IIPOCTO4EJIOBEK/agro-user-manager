# ОТЧЕТ: АРХИТЕКТУРА СИНХРОНИЗАЦИИ И МОЩНОСТЬ КЛАСТЕРА
## Дата: 25 февраля 2026

## 🔄 1. КАК РАБОТАЕТ СИНХРОНИЗАЦИЯ

Кластер построен по принципу "Master-Slave" для файлов и "Shared Storage" для контента:

### А. Синхронизация кода (PHP, JS, CSS):
- **Инструмент:** `lsyncd` (Live Syncing Daemon).
- **Процесс:** Веб-узел **10.0.1.220** является мастером. Любое изменение файла в `/home/bitrix/www/` мгновенно (задержка 15 сек) передается на узлы **10.0.1.221** и **10.0.1.222**.
- **Исключения:** Кэш, временные файлы и папка `upload` не синхронизируются через lsyncd, чтобы не создавать лишнюю нагрузку.

### Б. Общий контент (Папка /upload/):
- **Инструмент:** `NFS v4.2` (Network File System).
- **Процесс:** Папка `/upload/` физически находится на сервере Proxmox (на быстрых NVMe дисках) и подключена ко всем трем веб-узлам одновременно.
- **Результат:** Если пользователь загрузил аватарку на первом узле, она мгновенно доступна на всех остальных без копирования.

### В. Сессии и Кэш:
- **Инструмент:** `Redis` (на узле 10.0.1.210).
- **Процесс:** Все веб-узлы хранят сессии пользователей и кэш данных в оперативной памяти общего сервера Redis (выделено 58 ГБ RAM).
- **Результат:** При переключении пользователя между серверами (через HAProxy) "разлогинивания" не происходит.

## 🚀 2. СКОЛЬКО ЧЕЛОВЕК ПОТЯНЕТ КЛАСТЕР

На основе текущих ресурсов (Web: 3x16 CPU/32GB, DB: 128GB RAM, Redis: 64GB RAM):

| Режим нагрузки | Кол-во пользователей | Статус |
|----------------|----------------------|--------|
| **Комфортный** | **5 000** | ✅ Идеально. Запас по CPU ~60% |
| **Активный**   | **10 000** | ✅ Стабильно. Требуется мониторинг MySQL |
| **Пиковый**    | **15 000+** | ⚠️ Возможно замедление. Потребуются доп. веб-узлы |

### Почему кластер такой мощный:
1. **База данных в RAM:** Почти вся ваша БД (проект "prostory") помещается в `innodb_buffer_pool` (выделено 100 ГБ из 128 ГБ), что исключает медленные запросы к диску.
2. **Балансировка HAProxy:** Трафик равномерно распределяется между 3 мощными веб-серверами.
3. **NVMe Storage:** Папка `upload` на NVMe обеспечивает моментальное открытие документов и картинок.
4. **Оптимизированный PHP:** Настроены лимиты на 200 процессов на каждый узел (всего 600 одновременных PHP-запросов).

## 💡 РЕКОМЕНДАЦИИ
- **Правки вносить только на 10.0.1.220:** Изменения на 221 и 222 будут затерты автоматикой `lsyncd`.
- **Мониторинг:** Рекомендуется следить за количеством соединений MySQL в пиковые часы (лимит установлен в 5000 соединений).

**ИТОГ:** Текущая конфигурация является избыточной для 1000-3000 пользователей и уверенно держит **10 000 активных сотрудников**.
