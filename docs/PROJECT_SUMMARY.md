# Итоговый отчет по восстановлению кластера Битрикс24 "ПРОСТОРЫ"

## Описание проекта
В ходе работы над проектом был восстановлен доступ, настроена структура, синхронизация и исправлен ряд критических ошибок, блокирующих нормальную работу корпоративного портала Битрикс24 в кластерной среде. Была проведена диагностика и анализ взаимодействия компонентов, включая Nginx, HAProxy, Redis, БД MySQL и ноды Push & Pull (Node.js).

## 1. Решение проблемы истекшего демо-периода (Trial Reset)
*   **Проблема:** При попытке входа на портал появлялось сообщение "Срок работы пробной версии продукта истек" с блокировкой интерфейса.
*   **Решение:**
    *   Ключ был заменен на коммерческий (`P25-ML-PLBNQN7UM28BGK5XQMSI`, редакция Энтерпрайз 1000).
    *   Выполнено прямое вмешательство в базу данных (`b_option`) через скрипт `fix_license_2027.php`:
        *   `~mp24_paid` установлено в `Y`
        *   `~mp24_used_trial` установлено в `N`
        *   Дата оплаченного периода установлена до **2027 года** (`1807054400`).
        *   Удалены устаревшие значения `admin_passwordh` и `~demo_finish`.
    *   Лицензионный ключ `license_key.php` синхронизирован по всем трем веб-нодам (10.0.1.220, 221, 222).

## 2. Устранение фатальной ошибки "Unknown field definition UF_PUBLIC"
*   **Проблема:** При попытке открыть страницу структуры компании или начать диалог в мессенджере возникала фатальная системная ошибка `SystemException: Unknown field definition UF_PUBLIC`.
*   **Решение:**
    *   Написан скрипт `fix_uf_public.php`, восстанавливающий метаданные пользовательского поля.
    *   Поле добавлено в таблицу `b_user_field` с типом `boolean`.
    *   Добавлены переводы в `b_user_field_lang`.
    *   В таблицу физического хранения `b_uts_user` добавлена колонка `UF_PUBLIC`.
    *   Всем активным пользователям (более 300) принудительно установлено значение `UF_PUBLIC = 1`, чтобы они корректно фильтровались в модуле Интранет (список сотрудников) и мессенджере.
    *   Очищен ORM кэш (`Bitrix\Main\ORM\Entity::destroy('Bitrix\Main\User')`).

## 3. Настройка сервера очередей Push & Pull (WebSocket)
*   **Проблема:** Мессенджер выдавал ошибки "Вы не добавлены в этот чат", не обновлялись статусы прочтения и онлайн-присутствия. Сообщения о недоступности чата были следствием отвала Node.js сервера и кэша.
*   **Решение:**
    *   Извлечен актуальный Signature Key (`azTw0f...`) из конфигурационных файлов сервера 10.0.1.230.
    *   Переписана конфигурация модуля Push & Pull (через `fix_push_config.php`):
        *   Адрес публикации команд установлен напрямую в локальную сеть `http://10.0.1.230:8895/bitrix/pub/`.
        *   Адреса подписки клиентов перенаправлены на внешний домен через балансировщик: `wss://b24.ahprostory.ru/bitrix/subws/`.
    *   Используется сервер `sharded` (Bitrix Push server).

## 4. Синхронизация кластера и сброс кэшей (Redis)
*   Так как кластер состоит из 3 веб-нод за балансировщиком HAProxy, все конфигурации (`license_key.php`, `.settings.php`) и исправления БД применялись параллельно на все узлы.
*   Написан Ansible playbook (`bitrix24_sync.yml`) для поддержания идентичности конфигурации PHP-FPM, Apache и лицензий на всех нодах `[bitrix-web]`.
*   Решение большинства "плавающих" багов (особенно с видимостью `UF_PUBLIC` и триала) требовало полного сброса кэшей на сервере Redis `10.0.1.210` командой `FLUSHALL`.
